Single source of Truth

IMS collects and organizes information about all of InMobi's infrastructure from many different sources.

Clusters, Hosts, Data centers, Services, Racks, ... build and maintain structured information about infrastructural entities in one highly available distributed data store.

Describe

Describe Servers, Load Balancers, Clusters, Services and any thing you feel should be defined. Describe attributes and what values they can take. Describe who can do what, Describe just about everything. IMS leverages CouchDB to let you do this and even build your own user friendly modern web applications.

Load

Infra's workflows and SA's tools load metadata about infrastructure into IMS. Our Agent, which runs on all InMobi servers, contributes towards the 600+ pieces of facts that have been collected about all our assets. Additionally, you can load any other infrastructural assets information into IMS through a REST API for which numerous existing SDKs are available. Want even more control? Work with us and we will make additional tools and frameworks.

Consume

Use HTTP REST API to consume: check out IMS Search powered by the Collection API. There is even more: build custom application workflows around IMS or even get triggered as stuff happens to your beloved infrastructural assets. Orchestrate and Automate.

Replacing inefficiencies. One step at a time.

For reasons as old as Inmobi itself, our infrastructural information and tools have organised itself into siloes. Multiple twiki pages reference the state of our infrastructure, some arbitrary tools and maybe some puppet configuration files tell us the state of our information. Our approach is to constantly keep executing towards replacing each of these inefficient infrastructural touch points with Minimum Viable Products to help get bits in and out of IMS.

Oct 22-Dec 3, 2013

6 years of manually maintained cruft gets re-organized in 6 weeks.

There are things excel sheets and human minds cannot describe as effectively as what a single unix command can. Given we never had any single point of truth, we have all been sshing to machines to find such information. The easier way ought to be simpler: so we started a labourious project of discovering and keeping-in-touch with all of InMobi's assets by building an Agent and installing it everywhere. We knew this will not be easy, so we also ended up building a Reliability Dashboard to mainly help us get insights and run bug scrubbing activities to constantly improve IMS' data quality and reliability. At the time of this writing, about 70% of our assets are reliable. You can find if your assets are reliable or not and help improve it's reliability. Reliability Dashboard »

Excel Sheets uploaded to Twiki pages IMS.

On the Horizon...

People do unspeakable things go to IMS to find Load Balancer status of a host.

Today we tail log files to know whether a server has been taken out of rotation by the LB. This is not just painful, but just plain unclassy. This story, when completed, will enable IMS users to simply query an attribute of a host to know which LB / VIP it is behind and whether it is 'out' or 'in' according to the status recently polled from LB. You would be able to watch the lb_status field change from true to false within moments of failing the health check..

People have to query IMS HTTP API can use a shiny search and reports UI.

We acknowledge that exposing CouchDB API doesn't just cut it. We want to search in ways we all feel it must be searchable... and perhaps even embed a search result into emails or into dashboards. So, we want to build a user friendly search and reports UI to let users search for assets by providing complex search rules.
